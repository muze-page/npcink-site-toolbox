### 2. 存在的潜在问题与风险

尽管架构优秀，但在维护如此庞大的功能集合时，仍有一些细节可能会成为未来的技术债：

- **缓存失效（Cache Invalidation）的边界情况**：文档提到路由表对象缓存有 1 小时的 TTL。如果用户在后台开启了一个新功能，他们通常期望在前台立即看到效果。如果没有在保存配置时强制触发 `clear_cache()` 并重建缓存，这 1 小时的延迟会导致大量的用户报错反馈。
- **前端特效对 Core Web Vitals 的影响**：功能清单中包含了大量前台视觉特效，如“7 种点击特效”、“8 种背景特效”和“全站变灰”等。虽然这些是按需加载的，但引入过多的 DOM 操作和 Canvas/CSS 动画极易引发性能问题（特别是移动端），影响站点的 LCP 和 CLS 等核心网页指标。
- **第三方生态与主题耦合脆弱性**：插件集成了 B2 主题专属功能（商城统计等）以及大量国内生态（百度推送、微信验证、各大对象存储 OSS）。第三方 API 的变动或 B2 主题的更新，随时可能导致插件的某几个模块突然崩溃，维护这些外部依赖会耗费极大精力。
- **大一统（God Plugin）的维护噩梦**：90+ 功能集中在一个代码库中，虽然你做了模块化接口契约（`MaBox_Module_Interface`），但在实际迭代中，不同功能的 Bug 修复、测试覆盖会导致发版极其频繁。

**架构与工程化层面**

## 三、安全问题：文档写得对，但需要落到“每个接口、每个字段”

你文档里写了 SQL prepare、nonce、current_user_can、输出转义、输入清洗，这是正确方向。

但安全不能只写成“统一规范”，必须变成每个接口的强约束。

### 1. Public Nonce 不是安全边界

你有公开接口：

- 搜索日志
- 反爬虫验证
- 文章评分
- 微信解锁验证

文档里写公开接口是 `__return_true + nonce`。这个设计要谨慎。WordPress 官方 REST API 要求每个自定义路由都设置 `permission_callback`，公开接口可以用 `__return_true`，但这只表示“允许公开访问”，不等于安全。([WordPress Developer Resources][1])

**整改：公开接口必须加 5 件事：**

| 项目     | 建议                                         |
| -------- | -------------------------------------------- |
| 频率限制 | IP + User Agent + Cookie + 用户 ID 组合限流  |
| 行为限制 | 单文章评分、搜索日志、验证码验证都要限制次数 |
| 数据限制 | 不允许提交任意长文本、任意 URL、任意数组     |
| 来源检查 | 可选检查 referer/origin，但不能只依赖它      |
| 日志审计 | 记录异常频次、失败原因、IP、接口名           |

### 2. REST API 参数必须有 schema / validate / sanitize

文档里提到输入清洗，但没有看到你对每个 REST 参数做 `args` 级别的 `sanitize_callback` / `validate_callback` 约束。WordPress 官方文档建议在注册 REST endpoint 时为参数定义 `required`、`sanitize_callback`、`validate_callback`，让主回调只处理已经安全的数据。([WordPress Developer Resources][1])

**整改示例：**

```php
register_rest_route('mabox/v1', '/performance/db/clean', [
    'methods'  => WP_REST_Server::CREATABLE,
    'callback' => [$this, 'clean_database'],
    'permission_callback' => function () {
        return current_user_can('manage_options');
    },
    'args' => [
        'targets' => [
            'required' => true,
            'validate_callback' => function ($value) {
                $allowed = ['revisions', 'auto_drafts', 'trash_posts', 'spam_comments'];
                return is_array($value) && empty(array_diff($value, $allowed));
            },
        ],
        'dry_run' => [
            'default' => true,
            'sanitize_callback' => 'rest_sanitize_boolean',
        ],
    ],
]);
```

你现在要避免一种情况：**接口外层有 nonce，但字段内部可以传危险参数。**

## 四、权限问题：`manage_options` 不能解决所有场景

你文档里后台配置接口使用 `manage_options`，内容编辑使用 `edit_posts`，方向正确。 WordPress 的角色与能力体系本来就是用来控制用户在后台能看什么、做什么。([WordPress Developer Resources][2])

但你的插件功能太多，应该做更细的权限矩阵。

建议你新增一套内部能力映射：

| 操作             | 推荐 capability                                             |
| ---------------- | ----------------------------------------------------------- |
| 修改插件全局设置 | `manage_options`                                            |
| 批量替换文章     | `edit_others_posts` 或 `manage_options`                     |
| 数据库清理       | `manage_options`                                            |
| 数据库表导出     | `export` + `manage_options` 二选一，但建议 `manage_options` |
| 媒体 Alt 修复    | `upload_files` + `edit_posts`                               |
| 百度推送         | `publish_posts` 或 `manage_options`                         |
| AI 审核设置      | `manage_options`                                            |
| 查看日志         | `manage_options`                                            |
| 工单/反馈查看    | `manage_options` 或自定义 capability                        |

**不要简单认为“能进后台就是可信用户”。编辑、作者、运营人员都可能误操作。**

## 五、数据库表导出是当前最高风险功能之一

文档里有 `/mabox/v1/tools/tables` 和 `/mabox/v1/tools/table-data`，用于获取数据库表列表和导出 CSV。

这个功能对站长有用，但非常危险。

要开放所有表，至少要做：

1. 默认关闭。
2. 开启时显示高风险确认。
3. 只允许超级管理员。
4. 多站点下只允许 network admin。
5. 表名必须白名单，不允许前端传任意表名。
6. 字段级脱敏，比如 email、phone、token、password、secret。
7. 大表分页导出，不能一次性读全表。
8. 导出行为写入审计日志。
9. 导出的 CSV 文件不要落地到 public 目录。
10. 导出后自动删除临时文件。

这个功能如果处理不好，是最容易出安全事故的模块。

## 六、配置系统方向对，但要处理 autoload 和缓存失效

你把配置拆成多个 option，这是对的。文档里还有合并配置、导入导出、迁移、快照、回滚和 active modules 缓存。

但有两个隐患。

### 1. 大配置不要 autoload

如果你用 `update_option()` 默认写入，很多配置可能会进入 autoload。90+ 功能、多套快照、用户反馈、日志、AI 配置、OSS 配置，一旦 autoload 膨胀，会拖慢全站每次请求。

**整改：**

- 大配置：`autoload = false`
- 快照备份：`autoload = false`
- 日志/统计：不要放 option，放自定义表
- active_modules：可以 autoload，但必须很小
- merged_config 缓存：可以对象缓存，但不要无限膨胀

### 2. active_modules 缓存必须有严格失效机制

文档里写 active modules 存 option，并写对象缓存 1 小时。

问题是：**功能启停、主题切换、插件升级、配置导入、缓存插件清理、多站点切换时，都可能导致路由表过期。**

建议你在这些时机清除或重建 active_modules：

```php
update_option 后
import_config 后
restore_snapshot 后
switch_theme 后
upgrader_process_complete 后
activated_plugin / deactivated_plugin 后
插件版本迁移完成后
```

## 七、模块系统对，但静态类模式后期会拖累测试和扩展

你的模块规范是静态类 + `run($config)`，并通过注册表加载。这个 MVP 阶段可以接受。

但后期模块越来越多，静态类会带来几个问题：

1. 难做依赖注入。
2. 难做单元测试。
3. 难在运行时卸载 hook。
4. 静态状态容易污染。
5. 模块之间互相调用会越来越乱。
6. 命名空间缺失时容易和其他插件冲突。

**建议下一阶段不要马上大改，但新模块要逐步改为实例化模块：**

```php
interface MaBox_Module_Interface {
    public function register(): void;
    public function is_enabled(array $config): bool;
    public function get_hooks(): array;
}
```

模块加载器负责：

```php
$module = new Baidu_Push_Module($config, $logger, $settings);
if ($module->is_enabled($config)) {
    $module->register();
}
```

短期可以保留旧模块静态模式，但新模块建议用实例化写法。这样以后你做测试、日志、依赖注入、云端能力接入都会轻松很多。

## 十二、给你的最终建议

**你现在这版可以继续做，但下一步不应该是继续加功能，而是做“收敛版”。**

我的建议是：

> **v2.4 不加大功能，只做安全、权限、回滚、日志、限流、文档。**
> **v2.5 再做新功能和商业化入口。**

最重要的整改顺序是：

1. 先审 REST API。
2. 再审数据库/批量操作。
3. 再做高风险功能二次确认和 dry-run。
4. 再做日志中心。
5. 再做功能分层。
6. 最后补文档站和新手引导。
